解説と回答例

この課題のねらい

この課題のねらいは、「n8nの画面に慣れ、ノードを並べて線でつなぐ操作を自分の手で行うこと」です。

動画を見ているだけだと「分かった気」になりますが、実際にキャンバスを操作してみると、細かな設定項目や変数の受け渡しなど、見ているだけでは気づかない点がたくさん出てきます。小さくても自分で動かしたフローが1つあることが、セクション3以降の土台になります。

n8nの3つの基本要素

n8nのワークフローは、以下の3つの要素で成り立っています。

  • ノード（担当者）：1つ1つの処理を担当する四角い箱。「メールを取得する」「AIに要約させる」「Slackに投稿する」など、各ノードが役割を持つ。
  • ライン（連絡網）：ノード同士をつなぐ線。前のノードで作ったデータを、次のノードが受け取れるようにする「パイプ」。
  • トリガー（合図）：ワークフローが動き出すきっかけ。「手動で実行（Manual Trigger）」「毎日9時になったら（Schedule Trigger）」「メールを受信したら（Gmail Trigger）」など。

この3つの組み合わせで、どんな業務フローも表現できます。S2では最もシンプルな「Manual Trigger → いくつかの処理ノード → 分岐」の形で基礎を掴みます。

If と Switch の使い分け

分岐ノードは2種類あります。どちらを使うかは「分岐先がいくつか」で決まります。

  • If（イフ）：分岐が2つのときに使う。条件がtrue（真） か false（偽） のどちらかに処理が流れる。例：「時刻が12時より前か？」「金額が1万円以上か？」
  • Switch（スイッチ）：分岐が3つ以上のときに使う。1つの値を複数の条件で振り分ける。例：「天気が晴れ／雨／曇りのどれか？」「評価がA／B／Cのどれか？」

迷ったら：2分岐ならIf、3分岐以上ならSwitch。Ifをネストさせて3分岐にすることもできますが、Switchの方が読みやすく保守しやすいです。

回答例：時刻で挨拶を分岐するフロー

最もシンプルな例として、「現在時刻を見て挨拶文を変える」フローを組んでみます。ノード5個・分岐1つの構成です。

ノード1：Manual Trigger 　役割：手動でフローを起動する（キャンバス下部の「Execute workflow」ボタンで動く） ノード2：Edit Fields (Set) 　役割：現在時刻をデータにセットする 　設定：nowという名前のフィールドに式 {{ $now.hour }} を入れる 　（$now はn8nが用意している「今の時刻」を表す変数） ノード3：If 　役割：ノード2でセットした時刻が12より小さいかを判定する 　設定：条件 「{{ $json.now }} < 12」 　分岐：条件が真なら true側、偽なら false側に進む ノード4：Edit Fields (Set) ──── Ifのtrue側に接続 　役割：「おはようございます」という挨拶文をセット ノード5：Edit Fields (Set) ──── Ifのfalse側に接続 　役割：「こんにちは」という挨拶文をセット

実行すると、現在時刻に応じてどちらか一方のノード5が実行され、最終的に1つの挨拶文が出力されます。

よくあるつまずきどころ

  • ノードが線でつながっていない：ノードの右端の「○」から次のノードの左端の「○」へドラッグして線を引きます。線がないと次に進みません。
  • Ifノードの分岐先が逆：Ifノードは true（上） と false（下） に分かれます。どちらに何をつなぐかで結果が変わります。
  • 変数名のミスマッチ：前のノードで `now` という名前でセットしたのに、次のノードで `time` と参照しているとデータが拾えません。ノードの出力タブで実際のデータ名を確認しましょう。
  • Executionsに何も残らない：Manual Trigger から実行していない／保存していないワークフローは Executions に記録されない場合があります。一度「Save」してから実行してください。

なぜ「小さく動かす」ことが大事なのか

本格的な業務ワークフローは、トリガー・外部API・AI呼び出し・エラー処理など多くの要素が絡み合います。最初からそれを組もうとすると、どこで詰まったか分からなくなります。

まずは内蔵ノードだけで完結する最小のフローを動かし、「ノードがつながる」「データが流れる」「分岐する」という骨格を体感しておくと、セクション3以降で要素が増えても全体像を見失わずに済みます。

実務での活用イメージ

  • S3で「要件定義からの実装」に進む：S1で作った要件定義書をベースに、n8nで実装するフローを作ります。S2で学んだノード・ライン・分岐の操作が土台になります。
  • S4〜S5で「外部APIと繋ぐ」に進む：今回は内蔵ノードだけでしたが、Gmail・Slack・Gemini などの外部サービスに繋ぐときも、基本は「ノードを並べて線でつなぐ」の延長です。
  • S6で「デバッグ」を学ぶ：今回エラーで止まった受講生は、その時点でS6の予習になっています。Executions画面の見方や、エラーメッセージの読み方を詳しく扱います。
解説を読んでもいまいちわからない場合は、AIに質問してみましょう。ノードの使い方や、エラーの解決方法について具体的なアドバイスがもらえます。
