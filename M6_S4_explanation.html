解説と回答例

この課題のねらい

この課題のねらいは、「AIエージェントに"手足"を与える」体験をすることです。

セクション3までのワークフローは、n8nの内蔵ノードだけで完結していました。つまり、AIエージェントは「自分の頭の中で考える」ことしかできない状態でした。セクション4では、外部のデータを取ってくる／外部に働きかける手段を学びます。これでエージェントは初めて「現実の世界で役に立つ」存在になります。

APIとは（ざっくり）

API（エーピーアイ）は、異なるアプリ同士がデータをやり取りするための「専用窓口」です。動画1で紹介したレストランの例えがわかりやすい：

お客さん（n8nワークフロー） ↓ メニューを見て「ハンバーグください」と注文 注文係（API） ↓ 厨房に伝票を通す シェフ／厨房（サーバー） ↓ 料理を作る 注文係（API） ↓ 料理を客席に運ぶ お客さん（n8nワークフロー）が結果を受け取る

n8nのHTTP Requestノードは、この「注文係に伝票を渡す」役割です。指定したURLに指定した形式で問い合わせると、外部サービスからデータが返ってきます。

Webhookとは（APIとの違い）

APIが「こちらから取りに行く」のに対し、Webhookは「向こうから押し込まれる」仕組みです。

  • API（取りに行く）：n8nから「天気を教えて」と問い合わせる
  • Webhook（受け取る）：Slackでメッセージが投稿されたとき、n8nに自動で通知が飛んでくる

両者は相互に補完する関係です。「毎朝定時にAPIを叩いてニュースを取得する」「Slackにメンションがあったら即座に反応する」など、使い分けることでイベント駆動のエージェントが組めるようになります。

Credentials（認証キー）の安全管理

APIキーは「家の鍵」「銀行のカード」と同じです。漏れると他人に自分のアカウントで勝手に使われます。従量課金系のAPIだと、高額請求が来るリスクもあります（過去に実例あり）。

やってはいけないこと：

  • ノードのURL・Header・Body にキーを直書きする（ワークフローをエクスポートすると平文で流出）
  • キーをSlackやGitHubに貼り付ける
  • スクリーンショットにキーが映ったまま共有する

n8nの安全な使い方：

  • 必ずCredentials機能にキーを保存し、ノードからは「Credentialsを選択」して参照する
  • 従量課金系APIは利用上限を先に設定する（OpenAIなら月額上限。動画3の画面収録を参考に）
  • キーを誤って公開してしまったら、即座に発行元サイトでRevoke（無効化）して再発行する

回答例：OpenWeatherMap + Gemini で朝礼ブリーフィング

フロー構成（動画4と同じ題材）

1. Manual Trigger（手動起動） 2. HTTP Request：OpenWeatherMap API に「Tokyo の現在の天気」を問い合わせる - URL: https://api.openweathermap.org/data/2.5/weather - Query: q=Tokyo, units=metric, lang=ja - 認証: Credentials「OpenWeather_Key」を参照（キーは appid として自動付与） 3. Gemini：取得したJSONを元に「朝礼用のブリーフィング」を生成 - プロンプト例： 「以下の天気データを元に、朝礼で読み上げる簡潔な 一言ブリーフィング（3〜4文、敬体、服装の助言つき）を 作ってください。データ：{{ $json }}」 4. Set：最終メッセージを整形して出力

出力結果の例

おはようございます。本日の東京の天気は晴れ、気温は18度の予報です。 朝夕は冷えますので、薄手のジャケットがあると安心です。 日中は過ごしやすい陽気になりそうですので、外回りの方もどうぞ快適に。

S3まではダミーデータでこのフローを組んでいましたが、今回は本当に今の東京の天気が取れて、AIが実データから文章を生成します。この違いが「手足を得る」という言葉の実感に繋がります。

よくあるつまずきどころ

  • HTTP Request が401 (Unauthorized)：APIキーが正しくない／Credentialsが選択されていない／キーが無効化されている。発行元サイトで最新のキーを確認する。
  • HTTP Request が429 (Too Many Requests)：レートリミット超過。無料プランで呼び出し回数上限に達していないか確認。
  • 取得データが空 or 404：URLのパラメータ指定ミス。ドキュメントの例をそのまま貼って動くか確認する。
  • AIノードで「API key not set」：OpenAI/Gemini側のCredentialsが未設定。Credentialsを作ってノードに紐付ける。
  • AIに渡すデータが大きすぎてエラー：取得したJSON全体ではなく、必要な項目だけSetノードで絞ってからAIに渡す。
  • Webhookを使ったがTestモードで一度しか受信しない：Test URLは実行中の1回だけ有効。本番URLに切り替えるか、再度Test実行して送信する。

この課題で学んだことの広がり

  • API一覧が自分の武器になる：一度HTTP Requestの使い方を覚えれば、世の中の大半のAPIを叩けるようになる。社内APIにもつながる。
  • AIノードの位置づけが分かる：外部から取得した生データをAIに渡して「人が読める形」に変換する、という組み合わせが業務自動化の王道パターン。
  • セキュリティ感覚：API漏洩で青ざめる経験をする前に、Credentials機能を習慣づけておくのが長期的に一番効く。

実務での活用イメージ

  • 競合調査の自動化：プレスリリース配信サービスのAPIを叩いて競合キーワードで絞り込み、AIで「先週の競合動向3行サマリ」を生成してSlackに投稿
  • 顧客からの問い合わせフォーム対応：フォーム送信をWebhookで受けて、AIで問い合わせカテゴリを判定し、担当者のSlackにメンション
  • 社内データの定期レポート：社内APIで前日の売上／受注数を取得し、AIで「気になる変化」をコメント付きでレポート化
  • SNS監視：X(Twitter)・RedditなどのAPIで自社名を検索し、AIで感情分析／話題抽出
解説を読んでもいまいちわからない場合は、AIに質問してみましょう。API連携のエラー対処やCredentialsの設定方法について具体的なアドバイスがもらえます。
