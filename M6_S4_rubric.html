ルーブリック（採点基準）

採点基準

合計30点満点で評価します。評価対象は外部APIに実際に繋がったか、AIで加工できたか、認証情報を安全に管理したか、振り返りを書いたかの4点です。選んだAPIの種類・AIの出力文の文学的品質は評価しません。

観点 OK（満点） 惜しい（半分程度） NG（0点）
①外部API連携
（10点）		 HTTP Request（または Webhook）ノードで本物の外部APIと通信し、実行スクショで該当ノードが緑のチェックになっている。取得データが出力結果に反映されている。 HTTP Requestノードは置かれているが実行で赤エラー、または取得したデータが後続ノードで使われていない。 HTTP Request／Webhookノードが使われていない、またはSetノードで擬似化したまま（S3と同じ構成）。
②AI処理
（8点）		 Gemini または OpenAI ノードを使い、取得したデータを要約／整形／判定など何らかの加工をしている。出力結果がテキストで提出されている。 AIノードは置かれているが、出力が単にデータをオウム返しで返しているだけ／出力結果が未提出。 AIノードが使われていない。
③認証管理
（7点）		 APIキーがCredentials機能に保存されており、ノードからCredentials参照で使われている。使用Credentialsの名前とタイプが提出テンプレに明記されている。実行スクショにAPIキーの値が映り込んでいない。 Credentialsは使っているが名前／タイプの記載が不十分、またはスクショにキーの値が映ってしまっている（セキュリティ上は重点減点）。 APIキーがノードのURL・Header・Body に直書きされている／Credentialsを使っていない／Credentialsの記載が一切ない。
④振り返り
（5点）		 3〜5行で、S3の擬似フローと比べて何が変わったか／どこでつまずいたか／業務にどう活かせそうかが自分の言葉で書かれている。 1〜2行の感想のみ／「勉強になった」などの抽象的な記述のみ。 未記入。

採点の注意事項

選んだAPIの種類は評価しません。天気、ニュース、技術記事、公開データなど、受講生が選んだAPIが何であれ、実際に動いていれば可とします。

Webhook型で作った場合も同条件で評価します。「外部との実通信が成立している」ことが核なので、取りに行く（HTTP Request）／受け取る（Webhook）のどちらでも満点対象です。

AIの出力文の品質は評価しません。プロンプトの工夫で出力が改善できそうな場合はフィードバックで伝えるに留め、採点は「AIノードが機能しているか」で判断します。

セキュリティ違反は重点減点対象です。特に実行スクショにAPIキーが映っている／キーを直書きしている場合は、他の項目が優秀でも③は0点とし、キーの再発行を促してください。

フィードバックのポイント

高評価のケース：
HTTP Requestで実APIに接続できており、AIで意味のある加工がされていて、Credentialsの名前とタイプが明記され、実行スクショで全ノード緑のチェック（かつAPIキーが映り込んでいない）、振り返りで「S3の擬似フローとの違いを実感として書けている」場合、満点に近い評価です。Schedule Triggerで毎日定時実行にしている提出はボーナス的に高く評価してもOKです。

改善が必要なケース：
401/429などのHTTPエラーで止まっている場合は、「エラーメッセージをAIに貼って原因を相談してみてください。多くはキーの設定ミスかレート制限です」とフィードバック。AIの出力がオウム返しになっている場合は、「プロンプトに『〇〇形式で要約して』『〇文字以内で』など明示的な加工指示を入れると出力が変わります」と具体的に促します。

再提出を求めるケース：
APIキーがノードに直書きされている場合は、セキュリティ上必ず再提出を依頼してください。「まず発行元サイトで現在のキーをRevoke（無効化）→ 再発行 → Credentials機能で登録 → ノードから参照、の順で直してください」と手順を伝えます。
