解説と回答例

この課題のねらい

この課題のねらいは、「部分の動作確認」から「全体の通し動作」に視点を広げることです。

ここまでのセクションでは、ノード単位・処理単位で「動くこと」を確かめてきました。S5では、定時に自動で動き出し、情報を集め、AIで加工し、配信するという一連の流れを、1本のワークフローとして完結させます。これができると、作ったエージェントは業務の一部として本当に回り始めるようになります。

AIエージェントの4要素

業務で使われているAIエージェントは、だいたい以下の4つのブロックでできています。

  • ① トリガー（起動の合図）：いつ動き出すか。Schedule（定時）／Webhook（イベント受信）／Manual（手動）など。
  • ② 情報収集（インプット）：どこから何を取ってくるか。RSS／API／スプシ／メール受信箱など。
  • ③ AI処理（加工）：取ってきた情報を要約・分類・抽出・判定する。Gemini／OpenAIノード。
  • ④ 配信／記録（アウトプット）：人が使える形にして届ける／記録する。Slack／Gmail／スプシ／X／DBなど。

S5では、この4要素すべてを1本に繋いでエージェントを完成させます。どれか1つでも欠けると「業務で役立つエージェント」にはなりません。

RSSを情報源に選ぶ理由

動画1〜2で RSS を題材にしたのには理由があります。

  • 認証が要らない／ほぼ無料：公開RSSはキーなしで誰でも取得できる
  • 形式が標準化されている：サイトごとに書き方が違うWebページと違って、RSSはフォーマットが決まっている
  • 更新差分が取りやすい：最新記事だけを抜き出す処理が簡単
  • AIと相性が良い：タイトル・本文抜粋がテキストで得られるので、そのまま要約に回せる

同じパターン（定時起動 → 情報取得 → AI加工 → 配信）は、API／社内データベース／メール／ファイルシステムなど、他の情報源にもそのまま応用できます。RSSは"練習用"として理想的な入り口です。

結合テストとは

結合テスト（Integration Test）は、部分ではなく全体を通しで動かすテストのことです。ソフトウェア開発の世界では昔からある概念ですが、ワークフロー開発でも全く同じ発想が必要です。

ノードを1つずつクリックして「Output」タブを見て「動いた」と思っても、実際に全体を流すとノード間のデータ受け渡しで詰まることが頻繁にあります。例えば：

  • RSSノードは複数件の記事を返すが、AIノードは1件しか処理しない設定になっていた
  • AIの出力が期待するフィールド名と配信ノードの参照名がずれている
  • 途中でnull値が出て配信ノードが失敗する

これらは通しで実行して初めて見える不具合です。キャンバス下部の「Execute workflow」ボタンでエンドツーエンドを回す習慣をつけましょう。

回答例：TechCrunch → Gemini 3行要約 → Slack 配信

フロー構成

1. Schedule Trigger - 設定：毎日 9:00 AM JST に起動 2. RSS Feed Read - URL: https://techcrunch.com/feed/ - 最新5件を取得 3. Gemini（または OpenAI） - モデル：gemini-1.5-flash（軽量モデルで十分） - プロンプト： 「以下の記事タイトルと本文の一部をもとに、3行以内で要約してください。 1行目：何のニュースか 2行目：どう変わったか／何が起きたか 3行目：なぜ重要か 敬体で、専門用語を避けてください。 記事データ：{{ $json }}」 4. Slack（Post Message） - Channel: #ai-daily - Text: 「📰 今日のAIニュース\n\n{{ $json.summary }}\n\n🔗 {{ $json.link }}」 - Credentials: Slack_BotToken を参照

配信先 Slack に届く例

📰 今日のAIニュース OpenAIが推論特化の新モデル「o1-mini」を開発者向けに一般公開しました。 従来モデル比で数学・コーディングタスクが3倍速く動作し、コストも60%削減。 エージェント系プロダクトへの採用が加速する見通しです。 🔗 https://techcrunch.com/2026/04/18/...

よくあるつまずきどころ

  • RSSが複数件取れるのにAIが1件しか処理しない：AIノードが「Run Once for Each Item（各アイテムごとに実行）」設定になっていないことが多い。ノードの設定で切り替える。
  • RSSの本文が空：サイトによってはRSSにタイトル＋リンクだけで本文を含まないことがある。その場合は HTTP Request で記事URLを叩いて中身を取得する or タイトルだけで要約させる。
  • Slackで401エラー：Botトークンの権限不足。Slack App 設定で `chat:write` や `channels:join` を追加。
  • Schedule Trigger が時差で動かない：タイムゾーン設定に注意。n8n Cloudは UTC デフォルト／ Settings でJST に切り替え可能。
  • Googleスプシに書けない：スプシのOAuth認証が切れている or 対象スプシにアクセス権がない。Credentialsを再認証する。
  • 要約が3行を超える：プロンプトで「絶対に3行以内」「1行は50文字以内」など強い制約をかける。

"本番運用"に近づけるコツ

  • エラー時の通知：ワークフロー全体が失敗したとき、自分にDMが飛ぶようにErrorワークフローを設定する（S6で扱う）
  • 重複投稿の防止：前回取得した記事URLを Google Sheets に記録し、次回は差分のみ投稿する
  • 配信内容のA/Bテスト：プロンプトを少し変えた要約を数日並行運用し、チームのフィードバックで磨く
  • コスト監視：AIノードの呼び出し回数が増えると課金も増える。Schedule頻度と1回あたりの記事数でコスト試算しておく

実務での活用イメージ

  • 朝会の資料を自動で1枚：社内Wiki・営業数値・競合ニュースを集めて朝会用のSlack投稿を毎日8:45に
  • 採用の応募対応：応募フォーム（Webhook）→ AIで応募内容分類 → 担当者のSlackに自動アサイン通知
  • カスタマーサポート：問い合わせメール → AIで優先度判定 → 高優先度だけSlackの#alert に通知
  • 週次レポート：社内DBとSalesforceから数値取得 → AIでハイライト生成 → 経営層向けメールで毎週月曜配信
解説を読んでもいまいちわからない場合は、AIに質問してみましょう。エンドツーエンドで詰まった点や、配信先の設定について具体的なアドバイスがもらえます。
