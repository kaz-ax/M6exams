解説と回答例

この課題のねらい

この課題のねらいは、「エラーに出会ったときの動き方」を型として身につけることです。

動く自動化を作るスキルと、動かなくなったときに直せるスキルは別物です。業務で運用を始めるとエラーは必ず起きます。そのときに冷静にログを読み、AIに相談し、直せるかどうかで、任せられるエージェントの数が変わってきます。「動かしたまま放置して、いつの間にか3週間止まっていた」は避けたい事態です。

よくある停止原因4パターン

n8nのワークフローが止まる原因は、おおまかに以下の4種類に分類できます。

  • ① 認証系：APIキーの期限切れ・書き換え・権限不足（401 / 403）
  • ② 参照系：前ノードが返すデータと、次ノードが期待するフィールド名がずれる／`undefined`
  • ③ レート／制限系：APIの呼び出し回数上限（429）／無料枠の超過
  • ④ 外部サービス都合：相手側の一時障害（500系）／仕様変更で返り値が変わった

この課題では①と②を自分で起こして直す演習をします。③と④は実運用で出会ってから対処できるよう、今回はパターンとして頭に入れておきましょう。

ログ解析の3つの質問

Executionsのエラー画面を見るとき、以下3つを順番に自問すると原因に辿り着けます。

  1. どのノードで止まったか？（赤くなっているノードを特定する）
  2. そのノードに入ってきたデータは何か？（入力タブで前ノードの出力を確認）
  3. そのノードが期待するデータと、実際のデータはどこがずれているか？（パラメータ設定と照らす）

この3問に答えられたら、原因はだいたい特定できています。あとは直すだけ。

AIレスキューの使い方

エラーメッセージをそのままAIに貼り付けるだけでも助けになりますが、以下を添えると回答の精度が一気に上がります。

  • 使っているノードの名前（例：OpenAI Chat Model、Slack Send Message）
  • エラー全文（省略しない。Stack Traceがあればそれも）
  • 試したこと（Credentialsを確認した／別のノードに変えた等）
  • 「専門用語を避けて中学生にもわかる言葉で」の指示
  • 「修正のために確認すべき設定項目と順番を示して」の依頼

AIが的外れな回答を返してきた場合は「実際のワークフローはこうなっている」と現状を説明して差し戻す。これを2〜3往復すると、だいたい正解に辿り着きます。

エラーの起こし方

OpenAI Credentials「OpenAI_MyKey」のAPI Key欄を、末尾3文字だけ書き換えて保存。

エラーメッセージ（Executionsで見たもの）

ERROR: The resource you are requesting could not be found or the credentials provided are invalid. Full error: 401 Unauthorized { "error": { "message": "Incorrect API key provided: sk-...xyz", "type": "invalid_request_error", "code": "invalid_api_key" } }

ログ解析メモ

どのノードで：OpenAI Chat Model ノード どんなデータで：入力（RSSから取得した記事タイトル5件）はある。認証処理の段階で止まっている。 なぜ止まったか：401 Unauthorized かつ "invalid_api_key" とあるため、APIキーが無効。

AIレスキューのチャットログ（抜粋）

自分： n8nで以下エラーが出ました。OpenAI Chat Modelノードです。 「401 Unauthorized / invalid_api_key」。原因と修正方法を中学生にもわかる言葉で教えてください。 AI： これは「APIキー（OpenAIのパスワードのようなもの）が間違っている」ことを示しています。 確認手順： 1. OpenAIのサイトにログインし、API Keys画面で現在有効なキーを確認する 2. n8nの Credentials > OpenAI_MyKey を開き、キー欄を貼り直す 3. 「Test」ボタンで緑になるか確認 4. ワークフローを再実行 キーが漏洩した疑いがある場合は、OpenAIサイトで旧キーをRevokeしてから 新しいキーを発行してください。

修正内容のBefore/After

Before：Credentials「OpenAI_MyKey」のAPI Key末尾が`...xyz`（書き換えた誤値） After ：OpenAIサイトで現行キーを確認し、正しい末尾`...abc`で上書き保存 ※Test で緑確認、再実行でSuccess。

エラーの起こし方

Slack Send Messageノードの本文テンプレートを 「📰 {{ $json.title }}」から「📰 {{ $json.titleXYZ }}」に書き換え。

エラーの見え方

※このパターンはノードが赤くならないことも多い。 Slackに届いたメッセージが「📰 undefined」になってしまう。 Executionsは成功扱いだが、出力の中身が壊れる。 または厳密モードの場合： ERROR: Cannot read property 'titleXYZ' of undefined

ログ解析メモ

どのノードで：Slack Send Message ノード どんなデータで：前ノード（AI要約）の出力に`title`はあるが`titleXYZ`は無い なぜ止まったか：存在しないフィールド名を参照しており、undefinedがそのまま文字列化されている

AIレスキューのチャットログ（抜粋）

自分： n8nのSlackノードで投稿文が「undefined」と表示されます。 メッセージ本文には {{ $json.titleXYZ }} を指定しています。 AI： undefinedが出るのは「参照先に該当フィールドがない」ことを示します。 前ノードの出力を確認してください： 1. Slackノードの「Input」タブを開く 2. 実際に存在するフィールド名を確認（例：title, summary など） 3. メッセージ本文の参照を正しいフィールド名に直す 多くの場合、AIノードの出力は`json.message.content`や`json.choices[0]`の 配下にあります。Inputタブで確認するのが確実です。

修正内容のBefore/After

Before：「📰 {{ $json.titleXYZ }}」（存在しない参照） After ：「📰 {{ $json.title }}」（Inputタブで確認した実フィールド名）

デバッグが早くなる4つの習慣

  • エラーが出たら最初に"Input"タブを開く：前ノードが何を渡してきたかを確認するのが原因特定の最短路
  • エラーメッセージは省略せずAIに貼る：先頭だけ貼ると見落としが起きる。Stack Traceまで全部貼る
  • 直す前に「再現できる状態」を保つ：あれこれ触る前にスクショを残す。直してからBefore/Afterで振り返れる
  • 直したらSaveして再実行：直したつもりでも保存を忘れているケースが頻出。保存→再実行→Executions確認をワンセットで

コミュニティの使い分け

  • AI（ChatGPT / Gemini）：一般的なエラーや、よく出るパターンの解説には最速で役立つ
  • n8n公式フォーラム（community.n8n.io）：AIでは解決しない特殊ケース、n8n固有のバグ、新しいノードの使い方などに強い
  • n8n公式ドキュメント（docs.n8n.io）：ノードごとの仕様・パラメータの意味を正確に知りたいとき
  • GitHub Issues（n8n-io/n8n）：既知のバグや修正予定を調べるとき

実務での活用イメージ

  • 運用監視のためのエラー通知フロー：ワークフローが失敗したら自分にSlack DMが飛ぶ仕組みを別ワークフローで組む（Error Workflow設定）
  • エラー事例集の蓄積：社内で出たエラー・解決方法をNotionやWikiに溜めていくと、同じエラーに2度目会ったとき即解決できる
  • "仕様変更検知"のデバッグ：API側の仕様変更でいつの間にか動かなくなるケースに備え、出力の形式が変わっていないか月1でチェックする仕組みを作る
  • ペアデバッグ：詰まったらすぐ誰かにログを見せる／AIに聞く。1人で30分以上悩まない習慣づけ
解説を読んでもいまいちわからない場合は、AIに質問してみましょう。エラー解析の進め方や、よくあるつまずきのパターンについて具体的なアドバイスがもらえます。
