実習課題：意図的にエラーを起こしてデバッグする ─ エラー事例を1つ診断・修正しよう

目的

セクション5までは「動くものを作る」ことが中心でした。セクション6は「動かなくなったものを直す」スキルを身につけます。

業務で自動化を運用し始めると、エラーで止まることは絶対に避けられません。APIの仕様変更、認証キーの期限切れ、データ形式の変化、使っているサービスの一時的な不調 ── 理由は様々です。そのときに冷静に原因を特定して直せる力があるかどうかで、運用できるエージェントの数が変わってきます。

この課題の目的は、1つのエラーを意図的に発生させて、ログ解析 → AIレスキュー → 修正 → 再実行Success のデバッグの一連の流れを自分の手で1回転することです。

課題

セクション3〜5で作成した自分のワークフローを1つ選び、以下のエラーAまたはエラーBのどちらか1つを選んで意図的に発生させ、診断・修正してください。

  • エラーA（認証系）：ノードが参照しているCredentialsを無効な状態にして、実行時に認証エラー（401や403など）を発生させる
  • エラーB（参照系）：ノードのパラメータやプロンプトで、存在しないフィールド名を参照するように書き換えて、実行時に参照エラーまたは`undefined`を発生させる

どちらを選んでも採点上の有利不利はありません。自分が起こしやすい・修正イメージが湧く方を選んでください。両方やってみたい人はもちろんやってもOKです。

S5までで実際に遭遇したエラーがあれば、それを使ってもOK
セクション3〜5の課題中に実際にぶつかったエラーがあれば、それを事例としてそのまま充ててください（ログ・修正履歴が残っていれば再現不要）。エラーの種類がA／Bの分類に綺麗に収まらなくても、1つのエラーについて「診断 → 修正 → Success」まで追えていれば条件を満たします。

成果物

選んだ1つのエラーについて、以下6点を提出します。

  • エラー発生時のExecutions画面スクリーンショット ── 赤で止まっているノードが見える画面
  • エラー詳細ログ ── エラーメッセージの全文テキスト
  • ログ解析メモ ── どのノードで／どんなデータで／なぜ止まったかの自分なりの解釈（2〜3行）
  • AIレスキューのチャットログ ── エラーメッセージをAIに貼り付けて相談したやり取り（1〜3往復）
  • 修正内容のBefore/After ── どこを何に変えたか
  • 修正後のExecutions「Success」画面 ── 再実行で通ったことがわかる画面

手順

ステップ① デバッグ対象のワークフローを選ぶ

セクション3〜5で作成したワークフローから1つ選びます。S5の完成版（Schedule＋RSS＋AI＋配信）がベストですが、S3のインポート版やS4のAPI連携版でも構いません。

また、選んだワークフローはデバッグ演習で壊すので、複製（Duplicate）してから使うのがおすすめです。

ステップ② エラーを発生させる（A・Bのいずれか選択）

【選択肢A：認証系エラー】

ワークフローに含まれる任意のCredentials（AI／Slack／Gmail／スプシ／HTTP Request 系）を1つ選び、以下のいずれかで故意に無効化します。

認証系エラーの起こし方（どれか1つでOK）
  • Credentialsに保存されているAPIキーを1文字書き換える（保存後に"Test"でエラー確認）
  • ノードのCredentials選択を別の無効なCredentialsに切り替える
  • HTTP Requestノードで、Headerに仮の無効トークンを直接記入する（この課題中だけの例外、後でCredentials参照に戻す）

【選択肢B：参照系エラー】

AIノードのプロンプトや、Setノード、配信ノードの本文テンプレートなどで、存在しないフィールド名を参照するように書き換えます。

参照系エラーの起こし方の例
  • AIプロンプト内で {{ $json.title }} となっている箇所を {{ $json.titleXYZ }} のように別名に変える
  • Setノードのフィールドで、参照元に存在しない {{ $json.foobar.nonexistent }} を書く
  • 配信ノードのメッセージ本文で、前ノードにないフィールド名を指定する

AまたはBいずれかでワークフローを実行し、エラーが発生すること／Executionsでエラーとして記録されること（Bの場合は出力が`undefined`になること）を確認し、画面キャプチャを残します。

ステップ③ エラーを診断する（ログ解析）

Executionsから該当する実行をクリックし、赤くなったノードのエラーメッセージを確認します。以下3点をメモします。

  1. どのノードで止まったか（ノード名）
  2. どんなデータが原因か（入力タブ・出力タブを見る）
  3. エラーメッセージの内容（コピペしておく）

ステップ④ AIレスキューで原因を聞く

エラーメッセージをChatGPTまたはGeminiに貼り付けて相談します。

AIに渡すプロンプト例

n8nでワークフローを実行したら以下のエラーが出ました。原因と、n8n上での修正方法を専門用語を使わず中学生にもわかる言葉で教えてください。

【使っているノード】（例：OpenAI Chat Model、Slack Send Message など）
【エラーメッセージ】
（ここにエラー全文を貼り付け）
【試したこと】（あれば）

修正のために確認すべき設定項目と、具体的な直し方を順番に示してください。
AIで解決しない場合：コミュニティ活用
AIの回答で直らない場合は、動画4で紹介したn8n公式フォーラムや公式ドキュメントで同じエラーメッセージを検索してください。実際のユーザーが直した手順が見つかることが多いです。

ステップ⑤ 修正して再実行 → Success

  1. AIまたはコミュニティのアドバイスをもとに、該当ノードの設定を直す
  2. ワークフローを再実行
  3. Executionsで「Success」になることを確認
  4. スクリーンショットを撮って控える
